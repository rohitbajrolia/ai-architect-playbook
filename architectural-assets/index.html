    Architectural Assets¶

    Real building blocks for architects designing production AI systems.

    Everything here passed one test: would an architect actually reach for this when designing a real system?

    Decision Frameworks¶

    RAG vs Fine-Tuning¶

    Default to RAG. Override only when the evidence is clear.

    Situation Use Why
    Knowledge changes frequently — policies, prices, docs RAG Fine-tuning freezes knowledge at training time
    You need source citations or auditability RAG Retrieval gives you traceable context
    Fewer than 10,000 high-quality labelled examples RAG Fine-tuning needs volume to generalise
    Model needs to change reasoning style or tone Fine-tuning RAG cannot reshape how a model thinks
    Latency is critical and retrieval adds too much overhead Fine-tuning Baked-in knowledge has no retrieval step
    Domain vocabulary the base model does not know Fine-tuning + RAG Fine-tune for style, RAG for current facts

    Agents vs Deterministic Workflows¶

    Default to workflows. Add agents only when judgment is genuinely required that a workflow cannot provide.

    Situation Use Why
    Steps are known, order is fixed, inputs are predictable Workflow Cheaper, faster, easier to debug
    Path depends on intermediate results you cannot predict Agent Agents decide next steps based on what they find
    Errors need to be hard and immediately visible Workflow Agents can fail softly in ways that hide problems
    Task requires open-ended tool use or multi-step reasoning Agent Workflows cannot handle dynamic tool chaining
    Full audit trail of every decision is required Workflow Agent reasoning is harder to trace step by step
    Scale is high and cost must stay predictable Workflow Agent token usage is variable and can spike

    Choosing Evaluation Metrics¶

    Define what good looks like before you build. If you cannot define it before launch, you will not know if it is working after launch.

    What you care about Metric What it catches
    Answer correctness Exact match or F1 against ground truth Factual errors
    Retrieval quality Precision@K, Recall@K Wrong or missing context
    Faithfulness Answer grounded in retrieved context Hallucination
    Helpfulness Human preference or LLM-as-judge score Tone and completeness
    Latency P50 and P95 response time Slow paths under load
    Cost Tokens per request × price Budget drift

    Reference Architectures¶

    RAG Baseline¶

    The simplest RAG system that is still production-worthy.

    User query
→ Embed query (same model used at index time)
→ Vector search (top-K chunks scored by similarity)
→ Rerank (optional but recommended for precision)
→ Stuff into prompt with source metadata
→ LLM generates answer grounded in retrieved context
→ Return answer + sources to user

    Instrument: retrieval latency, chunk hit rate, faithfulness score, user feedback.

    What breaks first: stale index, chunk size mismatch, retrieval returning irrelevant context the model uses anyway.

    Agentic Workflow Baseline¶

    The minimum viable agent that is safe for production.

    User intent
→ Planner — LLM decides which tools to call and in what order
→ Tool executor — runs one tool at a time, checks output
→ State manager — tracks what has been done and what failed
→ Human approval gate — for actions that are hard to reverse
→ Result synthesiser — LLM summarises what was done
→ Return result + action log to user

    Instrument: tool call count per session, failure rate per tool, escalation rate, end-to-end latency.

    What breaks first: tools with side effects that cannot be undone, missing timeouts on long-running tools, agents retrying without a cap.

    Observability Loop¶

    What to log, what to score, and who reviews it.

    Every request logs:
→ Input (query or intent)
→ Retrieved context (chunks + scores)
→ Model output
→ Latency (retrieval, generation, total)
→ Token count
→ User feedback if available

Weekly review checks:
→ Retrieval precision trending down → reindex or tune chunking
→ Latency P95 growing → check vector DB load or chunk size
→ Hallucination rate up → tighten retrieval or add faithfulness check
→ Cost per request growing → check token bloat in prompts

    One person owns this review. If nobody owns it, it does not happen.

    Before You Ship — AI Feature Checklist¶

    Run this before releasing any AI feature into production.

    • Is the model version pinned so a provider update cannot silently change behaviour?
    • Is there an eval suite that runs on every deployment and blocks on regression?
    • Is retrieval freshness defined — how old can context be before it is considered stale?
    • Are all LLM calls wrapped with retry limits and per-call timeouts?
    • Is there a graceful fallback if the model or retrieval is unavailable?
    • Is PII excluded from prompts, context windows, and logs?
    • Is token and latency cost estimated at production-scale request volume?
    • Is there a named owner who reviews error rates and drift weekly?
    • For any irreversible action — is there a human approval step?
    • Are failure modes tested, not just happy paths?
